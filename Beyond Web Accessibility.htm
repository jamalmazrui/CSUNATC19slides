Follow Accessibility Best Practices

Follow Guidance on Applying WCAG 2.0 to Non-Web Information and

Communications Technologies (WCAG2ICT) - https://www.w3.org/TR/wcag2ict/

Follow platform recommended best practices [Notice the common themes]

Use native (platform) elements, controls, templates whenever possible

Make sure custom content provides appropriate accessibility

- Custom gestures/motion actuation have alternative options

(e.g., keyboard or UI Control) [WCAG 2.1 2.5.1A & WCAG 2.1 2.5.4A]

Web-based content pulled into native apps needs to follow WCAG 2.1 SC

Xbox One (UWP) Accessibility Overview

§   Xbox One apps are created under the Universal Windows Platform (UWP)

§   Support for keyboard interactions and screen readers.

§   Support for user customization, such as font, zoom setting (magnification), color, and high-contrast settings.

§   Alternatives or supplements for parts of your UI.

Joe recommends providing appropriate content structure using UIAutomation properties.

Source: https://docs.microsoft.com/en-us/windows/uwp/design/accessibility/accessibility-overview

Xbox One Accessibility Checklist

Set the accessible name (required) and description (optional) for content and interactive UI elements in your app.

Implement keyboard accessibility.

Visually verify your UI to ensure that the text contrast is adequate, elements render correctly in the high-contrast themes, and colors are used correctly.

Run accessibility tools, address reported issues, and verify the screen reading experience.

Make sure your app manifest settings follow accessibility guidelines.

Full checklist: https://docs.microsoft.com/en-us/windows/uwp/design/accessibility/accessibility-checklist

Google Android Best Practices

Label UI elements (*provide updated state information)

Group content

Group multiple related elements into on container for simpler navigation (e.g., button with icon & text)

Create easy-to-follow navigation

Make touch targets large

WCAG 2.1 2.5.5AAA recommends 44 X 44 CSS pixels. Android recommends 48dp X 48dp (Also, see Group Content above)

Provide adequate color contrast

Use cues other than color

Make media content more accessible

Apply ideas from accessibility resources

Activate accessibility settings & Test your app’s accessibility

*Ensure content changes and updates are discoverable * Joe’s recommendations not included in developer docs.

Compatibility Testing

§   Mostly about testing with the Available AT

(Unless you have direct access to source code)

§   When compatibility testing, just like in a desktop screen reader, testing needs to be done in all different modes

-               Xbox Narrator has a Controller & Narrator modes

-               Apple TV has Direct Touch, Follow Focus & Vertical Navigation - Android & iOS have reading and interaction modes

§   Need to test with Bluetooth or USB keyboard with AT on/off

*Not all platforms support keyboard navigation with AT off (e.g., iOS) § Turn on and off all accessibility setting (High contrast, captions, etc.)

Testing with

Screen Readers & Keyboard

Navigate page by semantic elements

- Use iOS/tvOS Rotor or Android Context menus

TAB, swipe, or arrow key through all elements

Set focus to each element to verify proper accessible name, role, value, and state.

Activate dynamic controls using screen reader and then keyboard only

Check for proper keyboard focus order and focus visibility

(not screen reader focus)

Enter blank and invalid values into forms to test error validation

Color Contrast testing

Test Hex values if known (most accurate)

Use screen capture applications or screenshot

-               Xbox One = Dev Home Remote Access

-               Apple TV = Xcode, QuickTime Player or LonelyScreen

-               Android = SCRCPY (Windows-Only) or Vysor

-               iOS = Xcode, QuickTime Player or LonelyScreen

Use color picker tool (Colour Contrast Analyser)

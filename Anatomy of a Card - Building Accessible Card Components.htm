Cards overview

What are cards?

A “card” in web design has many similarities to physical cards…

 

Like a credit card, it groups multiple elements on a single subject.

And like a deck of playing cards, it is often part of a larger collection.

What do cards look like?

Visually, cards are typically rectangular or square in shape (similar to a physical card) and contain one or more of the following elements:

•  Text

•  Images or icons

•  Media

•  Buttons

•  Links

Cards seem to be everywhere in web design…

Social media sites

News sites

Commercial sites

Why are card-based designs so popular?

Our designers explain…

“It helps model content into a format that is responsive” – CGI Designer Adriana Machado

 

“Card-based designs are:

•        Responsive

•        Mobile friendly

•        Intuitive for content organization

•        Logical visual

•        Best practice to display related information

•        Allow for progressive disclosure”       

– CGI Designer Joe Habib

Research suggests that card-based designs:

•        Grab users’ attention

•             Promote a clean, minimalist user interface, which many find to be aesthetically-pleasing

•        Provide a brief abstract of a topic to help the user decide whether they want to explore the content further (i.e. helpful entry points to more detailed webpages1)

Recap of cards overview

•        Cards group multiple elements (text, images, links, buttons, etc.) on a single subject. 

•        Cards can stand alone but are often part of a larger collection.

•        Cards are very popular in web design for various reasons, including their responsiveness and aesthetic appeal.

HTML semantics

How to code a card?

Determining how to code cards can be challenging since:

•        There is currently no standardized design pattern to provide consistent guidance on how to build card components.

•        The content contained within a card can influence how it is coded (i.e. a card containing only text and a link may involve different semantic markup than a more complex card with a heading and several calls-to-action).

 

As a result, developers may struggle in determining the semantic HTML markup they will use to build cards.

 

The first step…
Choosing the card container

Which HTML element is best?

Many of the card examples we have found have featured cards that are contained within <div> elements.

 

Our goal was to determine if there was a more meaningful native HTML element to group the contents within cards, such as the <section> and <article> elements.

What the HTML5 specification says

<section>

“The <section> element represents a generic section of a document or application. A <section>, in this context, is a thematic grouping of content. Each section should be identified, typically by including a heading (h1-h6 element) as a child of the section element.”

 

“A general rule is that the section element is appropriate only if the element’s contents would be listed explicitly in the document’s outline.”

“The <article> element represents a complete, or self-contained, composition in a document, page, application, or site. This could be a magazine, newspaper, technical or scholarly article, an essay or report, a blog or other social media post.”

 

“A general rule is that the article element is appropriate only if the element’s contents would be listed explicitly in the document’s outline. Each article should be identified, typically by including a heading(h1-h6 element) as a child of the article element.”

 

“The <div> element has no special meaning at all. It represents its children. It can be used with the class, lang, and title attributes to mark up semantics common to a group of consecutive elements.”

 

“Authors are strongly encouraged to view the <div> element as an element of last resort, for when no other element is suitable. Use of more appropriate elements instead of the <div> element leads to better accessibility for readers and easier maintainability for authors.”

<article>

<div>

Our recommendation

<section>

A <section> element is best when all of the content within it relates to one particular subject and the section belongs in the page/document’s outline (i.e. there is a heading element included within the content).

The <article> element is a specialized type of <section> element that groups content that could stand on its own and is therefore independently distributable or reusable (e.g., in syndication).1 An <article> element is best when the content within it is independent of its context.2

A <div> element is best when the content within it is not representative of a sub-section of the page that should be included in the page/document’s outline. A <div> element can be paired with another grouping element (e.g. an unordered list) to provide structure to the content.

<article>

<div>

Screen reader benefits for using <section> and <article>

<section> and <article> represent HTML5 sectioning elements. One benefit of using these elements over the <div> element is that they can be accessed using common screen reader shortcut keys for document regions. As a result, screen reader users can use the following keyboard shortcuts to quickly navigate to each card:

•        JAWS:

•        “R” key for <section>

•        “O” key for <article>

•        NVDA: “D” key

•        VoiceOver: Rotor

 

Screen reader users can also bring up a list of the landmarks or regions on the page to quickly navigate to the various cards on the page.

Labeling <section> and <article> elements

Terrill Thompson published an article on 2/14/2019 comparing the screen reader listening experience for <section> and <article> elements that are labeled vs. those that are not labeled. The article findings demonstrated a noted improvement (especially for <section>) when the elements were labeled.

Our demos leverage ARIA (Accessible Rich Internet Applications) markup to label the <section> and <article> elements for cards using the aria-labelledby attribute.

 

 

Cards model

Determining HTML semantics

Cards model overview

We came up with a cards model to help determine how to code card components. Our goal was to determine the best semantics to present both:

•     The type of content

•     Structure of the content

 

As a disclaimer, please keep in mind that this model might not perfectly fit your project’s specific requirements but should hopefully provide a starting point in considering how to build your card components.

Cards model illustrated

Cards model described

Does the content contain multiple elements that need to be grouped?

YES: The content represents a card. Move on to the next question.

NO: The content does NOT represent a card.

YES: The card represents a sub-section of the page and therefore we recommend using either the <section> or <article> element for the card container. Move on to the next question.

NO: The card does NOT represent a sub-section of the page and therefore should not be listed in the document’s outline. Use a generic <div> element for the card container.

YES: Use the <article> element for the card container, optionally with nested <header> or <footer> elements to organize the content.

NO: Use the <section> element for the card container with the aria-labelledby attribute defined to provide a unique accessible name.

Grouping cards into collections

Cards that are part of a repetitive pattern can be grouped together using list markup to create “card collections”.

 

Typically in card collections the order of the cards is not important and therefore unordered list markup (<ul>) can be used to group the cards into a list.

Full disclosure

We went back and forth on the semantics for Demo 3 in deciding between the <section> and <div> elements for the card container.

 

Ultimately, we landed on the <div> element because:

•        We didn’t feel that the individual card components represented sub-sections of the page that required heading markup and

•        During testing we didn’t find a significant difference in the functional performance.

Interaction patterns for card collections

Progressively disclosing cards

A card component is often part of a larger card collection.

 

When there is potential for a large volume of cards on a webpage, developers will often implement functionality such as a “Load More” button or pagination to progressively disclose cards to the user.

Load More functionality

One design pattern we are frequently seeing paired with a card collection is the use of a “Load More” button. This can be especially helpful for responsive pages where there is less screen real estate available.

 

For example, Demo 3 containing our product card collection displays a “Load More” button to allow the user to add another set of cards to the overall collection.

Load More accessibility considerations

We recommend implementing Load More functionality as follows to ensure that screen reader users perceive when new cards are added to the collection:

•        Once the “Load More” button is activated, fire an ARIA live region so that users hear that the button activation was successful and new cards have been added. For Demo 3, we defined ARIA role=“status” (which is an implicit live region) to the visible text “New products added” to create the live region.

•        Programmatically move focus to the start of the new card results so that users can immediately start reading the new results.

Pagination can also be used to divide large card collections into separate pages.

Testing considerations

Keyboard interactions

When tabbing through the interactive elements within a card, keyboard users should tab to each element within the first card before moving to the next card.

 

Verify that a keyboard user is able to:

•     Navigate to and operate all interactive elements

•     Navigate to interactive elements in a logical order

•     Visually track their cursor focus as they navigate through interactive elements (this is essential for sighted keyboard users)

 

Verify that the following does NOT happen:

•     Keyboard focus does not become “trapped” within a particular interactive element or series of interactive elements

•     Focus is not improperly redirected once an interactive element receives focus

Reading order

In terms of reading order:

•        All of the content contained within the first card must be read before the text in the next card to make sense to screen readers and other types of Assistive Technology (AT).

•        This reading order needs to be maintained in responsive views as well, as card collections are reflowed to fit the contents of a smaller screen.

•        It would NOT be acceptable if the reading order skipped back and forth between each of the cards (e.g., a screen reader reads the first line of the first card and then the first line of the second card) as this would render the content illogical.

 

 

 

 

 

 

 

Lessons learned

LL 1: Card borders disappear in HCM

Cards should adapt appropriately to various changes of presentation, including when Windows High Contrast Mode (HCM) or a user-defined style sheet is applied.

 

We’ve seen some instances in HCM where the styling that defines the card container “disappears” when HCM is applied and therefore the visual grouping of each card is no longer present.

 

This issue can be addressed by ensuring that a border CSS property is properly defined for the card container.

 

 

LL 2: Content not exposed for “flip” cards

Cards that “flip” to reveal additional content when focused can be tricky from an accessibility perspective.

 

In particular, we’ve seen instances where flip cards:

•        Only flip on mouse hover, but not on keyboard focus.

•        Show different content when focused vs. not focused, which means a screen reader user would need to navigate to the card using TWO different reading modes (i.e. line-by-line reading in addition to tabbing) in order to hear ALL of the card contents announced.

 

If flip cards will be used, there must be careful consideration during the design phase to ensure an accessible experience for all users.

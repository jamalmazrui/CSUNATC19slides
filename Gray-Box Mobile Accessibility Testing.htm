But, to date…

•    Ensuring the accessibility of a mobile application has, for some, been far from easy. 

•    With different Accessibility requirements needed on different platforms.

•    Along with different testing methods, skills, knowledge, tools and Assistive Technologies (AT).

•     With each test-suite coded in specific computer languages (swift; java). 

•     He ensures these tests are included along with the application’s own source code in a specially built “test” variant...

Bill’s (Fictional) Story…

FYI - White-Box Testing…

•    In a mobile testing context, white-box solutions generally consist of a set of separate iOS or Android tests which need to be included (“baked in”) with the source code in order to create a testable version of an application.

 

The Down-sides to White-Box Testing…

•    In more recent times we’ve seen the general market’s appetite for third-party “baked-in” solutions shrink.

•    Possibly driven by factors such as security concerns...

 

Down-sides of White-Box Testing cont’d…

White-box testing also requires a good level of accessibility testing knowledge in order to obtain meaningful / actionable results.

•     Developers need to know code-level accessibility requirements e.g. mechanisms that expose additional text descriptions to ATs.

•     Testers need to understand methods for testing the as-implemented accessibility features on the different platforms; and

•     All need an appreciation of the ways different ATs on different platforms deal with coded accessibility features.

 

 

A further down-side…

So, our aim…

To consider a new method of mobile accessibility testing:

•     applying so-called “gray-box” testing;

•     which only uses higher-level, cross-platform, tests; and the widely-used testing frameworks / tools used to run them; and

•     most importantly does not require any additional code to be “baked-in” to a specially built test application! 

So, you’ve probably heard of Black-Box testing…

•    Let’s say you have a box, which is sitting on the pre-scan conveyor belt of an airport luggage scanner…

•    The security guard looks at the outside of the box whilst it rolls up to the scanner. 

•    What they are doing is black-box testing: testing of a device, program or system whose workings are completely unknown.

 

And, of course White-Box testing…

•    When a box is flagged for opening, and the guard opens the box – then they are doing white-box testing: testing of a device, program or system whose workings are completely known.

 

 

Again using the security context…

So, what is gray-box testing?

•     When the box rolls into the scanner; and the guard looks at an x-ray representation of what is in the box. 

•     They are now doing gray-box testing: testing of a device, program or system whose workings are now partially known. 

•     Here the x-ray is simply a representation of what’s in the box; giving proxy knowledge of the box’s contents.

 

And, again, using the security context…

Accessibility testing…

•    Often A11y tests use Selenium commands to:

•     Move a web page in a browser to the correct view (technically Document Object Model [DOM] state);

•     then inject a JavaScript Accessibility Testing Library;

•     then execute the library’s accessibility tests against the current DOM state; and

•     finally extract the results back into the test program.

•    FYI – The DOM is the living representation of all of the UI components.

 

Again, in a web context using Selenium…

FYI - Testing in a mobile context…

•    For clarity, when mentioning testing in a mobile context – we’re talking about three different things:

•     testing UIs of native iOS or Android apps;

•     testing UIs of native hybrid apps (built for example with Cordova / PhoneGap);

•     testing UIs of mobile web apps – really web content in a mobile browser.

 

 

So now, let’s compare how gray-box testing works in a mobile context?

It works in a very similar manner:

•      Selenium Server is replaced with Appium Server.

•      Webdriver is replaced by a platform-specific app-driver.

•      Browser is replaced by Native App; but the mobile browser can also be tested against in Mobile Web Apps.

•      The DOM in the Browser is replaced by an XML representation of the native UI components in their current state, made available by the app-driver.

FYI - The XML representation is generated through a tree-branch-leaf traversal of the native object tree for the application.

FYI - Appium Server…

•    There are several QA tools which can be used to obtain this XML “x-ray representation,” however, here we’ll concentrate on Appium - a very popular free + open-source tool - although there are of course, many other great products.

•    Appium: is a test automation framework for use with native, hybrid and mobile web applications – it is “Selenium for Mobile”;

•     which can also be effectively used in conjunction with Continuous Integration (CI) servers, in support of Agile / Devops activities.

 

 

 

How gray-box testing works in a mobile context cont’d…

•     The ability for app-drivers to facilitate an XML representation of the native UI components in their current state is fundamental to our being able to conduct accessibility testing using this process. 

•     The XML representation is not the actual source code - just as the HTML DOM is not the source code of a web page. 

•     It is simply a representation of the UI at that point in time.  

•     But, as such, it’s really not too long a stretch of the imagination to think of this XML as an “x-ray representation” – from which we get proxy knowledge about the internal workings of the UI.

Testing accessibility in a mobile context

•     From an accessibility testing perspective, as long as this XML representation exposes the node types and attribute values we can test it as an adequate stand-in, or proxy, for the native UI itself.

•     And, although there are differences between the XML representation of an iOS UI, and an Android UI, when interpreted correctly, the exact same high-level accessibility tests can be used to test both.

 

 

 

A11y testing in a mobile context…

Often tests use “Selenium” commands to:

•     Move a native mobile UI to the correct view; then

•     Extract the XML representation;

•     Translate the XML;

•     Run the accessibility tests from an Accessibility Testing Library; and

•     Get results.

With a few differences, but not many, to the web context…

But, that’s not all…

•    WCAG 2.1 was started with the goal to improve accessibility guidance for three major groups: one of which is users with disabilities on mobile devices.

•    The question then? With gray-box testing can mobile application UIs be tested against WCAG 2.1?

Yes! WCAG 2.1 can be used to test native mobile UIs!!!

•    The reason being that WCAG 2.1 is platform agnostic, and deals with expected outputs to be tested for through the UI.

•    By careful translation of the XML gained through gray-box testing to HTML5 it is possible to test mobile-relevant Success Criteria from WCAG 2.1; and

•    Even more! The same tests from JavaScript Accessibility Testing Engines can be used!

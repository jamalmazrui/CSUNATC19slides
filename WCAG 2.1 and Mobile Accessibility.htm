WCAG 2.1

•     WCAG 2.1 builds on and extends WCAG 2.0, but does not supersede or replace WCAG 2.0.

•     WCAG 2.1 is now the official recommendation of the World Wide Web Consortium (W3C).

•     All WCAG 2.0 success criteria are still included in WCAG 2.1 with their same numbers.

WCAG 2.1 Focuses On Accommodating More User Groups

All 6 of the new success criteria that benefit those with motor and dexterity disabilities also benefit users of mobile devices.

 

Mobile devices need flexible orientation

Content Orientation

•     Content orientation should never be restricted, so that it will always match the device orientation

•     If the user has locked the device display to a specific orientation, the application should pick up that setting and display content accordingly

 

Exceptions

Anytime a particular display orientation is essential to the content.

 

 

Ensuring Compliance

•    Ensure that the device is not locked to a specific orientation

•    Rotate the device from one orientation to another

•     There are no layout specifications for the different orientations

 

•    Think about responsive design:

•     Use CSS to set the orientation to allow both landscape and portrait

•     Show/hide controls may be used to allow access to content in different orientations

 

 

Complex gestures can be difficult for some people

Single-Point Gestures

All operations must use simple gestures that require only a single touch.

 

 

 

 

Path-Based Gestures

•     Any interaction where the success of the gesture is dependent on the path:

•      Swiping

•      Tracing a specific path

•      Pinch-zoom

•      Split tapping

•     Excludes:

•      Drag-drop (not considered path based)

•      Gestures for built-in AT

•      Gestures defined by the OS or user agent

 

2.5.1- Exceptions

Any functionality that must be based on complex paths in order for the interaction to be meaningful.

 

•     Entering a signature

•     Free-form drag-drop actions

 

2.5.1 - Ensuring Compliance

Provide controls that do not require complex gestures and which perform the same function as complex gestures

Ensuring Compliance with 2.5.2

Anything activated by gestures should also be operable by UI components

Motion Actuation

Functionality that can be operated by device motion or user motion can also be operated by user interface components.

 

Motion Actuation (cont.)

•    Only concerns intentional movement or gesturing by the user

•    Does not include the movement of users through space

•     Geolocation

•     Beacons

•     Activity measurement

•    Does not include motion needed to operate a pointer, keyboard, or assistive technology

Ensuring Compliance with 2.5.5

Using both a touch screen and a Bluetooth keyboard for input

2.5.6 Exceptions

Any time a particular input is essential to the experience or to perform a task.

 

 

 

 

Ensuring Compliance with 2.5.6

•    Test your content with various input mechanisms, and ensure the user can switch between different input types at any time.

•    Developers:

•     Only use high-level and input-agnostic JavaScript event handlers:

•     focus

•     blur

•     Click

•     Register event handlers for keyboard and pointer inputs simultaneously in JavaScript
